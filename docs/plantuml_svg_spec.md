# PlantUML SVG Output Specification

Reference spec for replicating PlantUML's SVG output in RustUML.
Derived from analysis of 12,500+ golden SVG files generated by PlantUML 1.2026.3beta6.

---

## 1. Root `<svg>` Element

Every PlantUML SVG begins with the same root element structure:

```xml
<svg xmlns="http://www.w3.org/2000/svg"
     xmlns:xlink="http://www.w3.org/1999/xlink"
     contentStyleType="text/css"
     data-diagram-type="SEQUENCE"
     height="152px"
     preserveAspectRatio="none"
     style="width:145px;height:152px;background:#FFFFFF;"
     version="1.1"
     viewBox="0 0 145 152"
     width="145px"
     zoomAndPan="magnify">
```

### Attributes (alphabetical order in output):

| Attribute | Value | Notes |
|-----------|-------|-------|
| `xmlns` | `http://www.w3.org/2000/svg` | Always present |
| `xmlns:xlink` | `http://www.w3.org/1999/xlink` | Always present |
| `contentStyleType` | `text/css` | Always `text/css` |
| `data-diagram-type` | Diagram type string | See table below |
| `height` | `{N}px` | Matches viewBox height |
| `preserveAspectRatio` | `none` | Always `none` |
| `style` | `width:{W}px;height:{H}px;background:#FFFFFF;` | Inline style with background |
| `version` | `1.1` | Always `1.1` |
| `viewBox` | `0 0 {W} {H}` | Matches width/height |
| `width` | `{N}px` | Matches viewBox width |
| `zoomAndPan` | `magnify` | Always `magnify` |

### `data-diagram-type` values:

| Diagram | Value |
|---------|-------|
| Sequence | `SEQUENCE` |
| Class | `CLASS` |
| State | `STATE` |
| Activity | `ACTIVITY` |
| Component | `DESCRIPTION` |
| Deployment | `DESCRIPTION` |
| Use Case | `DESCRIPTION` |
| Object | `CLASS` |

**IMPORTANT**: Component, Deployment, and Use Case diagrams all use `DESCRIPTION`.
Object diagrams use `CLASS`. Some error/syntax-error SVGs omit `data-diagram-type` entirely.
The background color in the style is `#FFFFFF` for the default theme; error diagrams use `#000000`.

### Immediately after `<svg>`:

```xml
<?plantuml 1.2026.3beta6?>
<defs/>
<g>
  ... all content ...
  <?plantuml-src {encoded-source}?>
</g>
</svg>
```

- The `<?plantuml version?>` processing instruction appears immediately after the opening `<svg>` tag.
- An empty `<defs/>` element follows.
- All diagram content is wrapped in a single top-level `<g>` element.
- The `<?plantuml-src {encoded}?>` processing instruction appears as the last child of the top-level `<g>`, containing the source encoded (likely Deflate + Base64).

## 2. Element Wrapper Patterns (`<g>` groups)

PlantUML wraps diagram elements in `<g>` groups with semantic class names and data attributes.

### 2.1 Sequence Diagram Elements

#### Participant Lifeline
```xml
<g class="participant-lifeline" data-entity-uid="part1" data-qualified-name="Alice" data-source-line="1" id="part1-lifeline">
```

#### Participant Box (Head and Tail)
```xml
<g class="participant participant-head" data-entity-uid="part1" data-qualified-name="Alice" data-source-line="1" id="part1-head">
<g class="participant participant-tail" data-entity-uid="part1" data-qualified-name="Alice" data-source-line="1" id="part1-tail">
```

#### Message Arrow
```xml
<g class="message" data-entity-1="part1" data-entity-2="part2" data-source-line="1" id="msg1">
```

#### Activation Bar
Activation bars are rendered as standalone `<g>` groups with a `<title>` child (participant name), **not** with class/data attributes:
```xml
<g><title>Bob</title><rect fill="#FFFFFF" height="29.3105" style="stroke:#181818;stroke-width:1;" width="10" x="109.6558" y="67.7988"/></g>
```
Colored activations use the specified color as fill (e.g., `fill="#0000FF"` for `#blue`).

### 2.2 Class/Object Diagram Elements

#### Class Entity
```xml
<g class="entity" data-qualified-name="MyClass" data-source-line="1" id="ent0002">
```
- HTML comment `<!--class MyClass-->` precedes the `<g>` element.

#### Link/Relationship
```xml
<g class="link" data-entity-1="ent0002" data-entity-2="ent0003" data-link-type="dependency" data-source-line="3" id="lnk4">
```
- HTML comment `<!--link A to B-->` or `<!--reverse link A to B-->` precedes the group.

**`data-link-type` values:**
| PlantUML syntax | `data-link-type` |
|----------------|-----------------|
| `-->` (solid arrow) | `dependency` |
| `..>` (dashed arrow) | `dependency` |
| `--` (solid line, no arrow) | `association` |
| `<\|--` (inheritance/extends) | `extension` |
| `..\|>` (implements) | `extension` |
| `*--` (composition) | `composition` |
| `o--` (aggregation) | `aggregation` |
| `<-->` (bidirectional) | `dependency` |
| `<..>` (bidirectional dashed) | `dependency` |

### 2.3 State Diagram Elements

#### State Entity
```xml
<g class="entity" data-qualified-name="SS" id="ent0002">
```
Note: State entities sometimes lack `data-source-line`.

#### Start Pseudo-state
```xml
<g class="start_entity" data-qualified-name=".start." data-source-line="2" id="ent0002">
```

#### End Pseudo-state
```xml
<g class="end_entity" data-qualified-name=".end." data-source-line="3" id="ent0004">
```

#### Links
```xml
<g class="link" data-entity-1="ent0002" data-entity-2="ent0002" data-link-type="dependency" data-source-line="2" id="lnk3">
```

### 2.4 Component/Deployment/Use Case Elements

#### Container/Cluster
```xml
<g class="cluster" data-qualified-name="G1" data-source-line="1" id="ent0002">
```

#### Entity (Component, Node, Actor, Use Case)
```xml
<g class="entity" data-qualified-name="G1.AA" data-source-line="2" id="ent0003">
```
Nested entities use dot-separated qualified names (e.g., `G1.AA`).

#### Links
Same as class diagrams.

### 2.5 ID Numbering

- Entity IDs: `ent0002`, `ent0003`, `ent0004`, ... (starts at 0002, increments)
- Link IDs: `lnk3`, `lnk4`, `lnk5`, ... (starts at 3 in most cases, seems to continue from entity counter)
- Sequence participant IDs: `part1`, `part2`, `part3`, ...
- Sequence message IDs: `msg1`, `msg2`, `msg3`, ...
- Lifeline IDs: `{partN}-lifeline`
- Head/Tail IDs: `{partN}-head`, `{partN}-tail`

## 3. Styling Pattern

PlantUML uses **inline `style` attributes** for strokes and stroke-widths, and **direct attributes** for fills.

### Typical pattern for rectangles:
```xml
<rect fill="#F1F1F1" height="48" rx="2.5" ry="2.5"
      style="stroke:#181818;stroke-width:0.5;" width="87.1113" x="7" y="25.27"/>
```

### Typical pattern for lines:
```xml
<line style="stroke:#181818;stroke-width:0.5;" x1="8" x2="93.1113" y1="39" y2="39"/>
```

### Typical pattern for ellipses:
```xml
<ellipse cx="22" cy="23" fill="#ADD1B2" rx="11" ry="11"
         style="stroke:#181818;stroke-width:1;"/>
```

### Typical pattern for paths:
```xml
<path d="M..." fill="none" style="stroke:#181818;stroke-width:1;"/>
```

### Typical pattern for polygons:
```xml
<polygon fill="#181818" points="..." style="stroke:#181818;stroke-width:1;"/>
```

**Summary**: `fill` is always a direct attribute. `stroke` and `stroke-width` are always in the `style` attribute. This is consistent across all element types.

### Dashed lines:
```xml
style="stroke:#181818;stroke-width:0.5;stroke-dasharray:5,5;"    <!-- lifeline dashes -->
style="stroke:#181818;stroke-width:1;stroke-dasharray:7,7;"       <!-- dashed dependency arrows -->
style="stroke:#181818;stroke-width:1;stroke-dasharray:2,2;"       <!-- return/response message dashes -->
```

## 4. Text Rendering

All text elements follow this pattern:

```xml
<text fill="#000000" font-family="sans-serif" font-size="14"
      lengthAdjust="spacing" textLength="32.7236" x="12" y="25.5352">Alice</text>
```

### Standard attributes:

| Attribute | Common Values | Notes |
|-----------|---------------|-------|
| `fill` | `#000000` | Text color; always a direct attribute |
| `font-family` | `sans-serif` | Default; `monospace` for code/deprecation warnings |
| `font-size` | `14` (body), `13` (messages/notes), `12` (stereotypes), `10` (warnings) | Size varies by context |
| `font-style` | `italic` | Present for stereotypes and abstract class names |
| `font-weight` | `700` or `bold` | Present for titles, group labels |
| `lengthAdjust` | `spacing` | Always `spacing` |
| `textLength` | Precise float value | Pre-calculated text width in SVG units |
| `x` | Float | Horizontal position |
| `y` | Float | Vertical baseline position |

### Text size by context:
- **Class/entity names**: font-size `14`
- **Stereotype labels** (`<<entity>>`): font-size `12`, font-style `italic`
- **Sequence messages**: font-size `13`
- **Note content**: font-size `13`
- **Group/combo labels** (alt, loop, etc.): font-size `13`, font-weight `700`
- **Group guard text** ([ok], [fail]): font-size `11`, font-weight `700`
- **Cluster/package titles**: font-size `14`, font-weight `700`
- **Deprecation warnings**: font-family `monospace`, font-size `10`

### textLength Calculation:
The `textLength` attribute is a precise floating-point width calculated from the font metrics of the Java AWT rendering engine. Each character has a specific width that depends on the font and size. For exact match, the renderer must use the same font metrics (Liberation Sans or similar sans-serif font at the specified size).

### Special text decorations:
```xml
text-decoration="wavy underline"    <!-- used in syntax error displays -->
```

### HTML entities:
- `&#171;` = left guillemet (used in stereotypes)
- `&#187;` = right guillemet
- `&#160;` = non-breaking space

## 5. Class Entity Icons (Stereotype Circles)

Each class-like entity has a colored circle + letter glyph rendered as SVG paths.

### Icon structure:
```xml
<ellipse cx="22" cy="23" fill="#ADD1B2" rx="11" ry="11" style="stroke:#181818;stroke-width:1;"/>
<path d="M..." fill="#000000"/>
```

The ellipse is always `rx="11" ry="11"` (22px diameter), stroke `#181818`, stroke-width `1`.

### Colors by entity kind:

| Kind | Ellipse Fill | Letter | Notes |
|------|-------------|--------|-------|
| Class (C) | `#ADD1B2` (soft green) | C | Path is a "C" glyph |
| Interface (I) | `#B4A7E5` (soft purple) | I | Path is an "I" glyph |
| Enum (E) | `#EB937F` (soft red/pink) | E | Path is an "E" glyph |
| Abstract class (A) | `#A9DCDF` (soft cyan) | A | Path is an "A" glyph, name is italic |
| Annotation (@) | `#E3664A` (orange-red) | @ | Path is a "@" glyph |

## 6. Arrows/Edges

### 6.1 Sequence Diagram Messages

#### Synchronous message (solid line, filled arrowhead):
```
A -> B : message
```
```xml
<polygon fill="#181818" points="97.6558,63.7988,107.6558,67.7988,97.6558,71.7988,101.6558,67.7988"
         style="stroke:#181818;stroke-width:1;"/>
<line style="stroke:#181818;stroke-width:1;" x1="28.3618" x2="103.6558" y1="67.7988" y2="67.7988"/>
```

**Arrowhead polygon pattern** (pointing right):
```
tip_x-10, tip_y-4
tip_x,    tip_y
tip_x-10, tip_y+4
tip_x-6,  tip_y
```
The arrowhead is a filled diamond-shaped polygon at the tip. For left-pointing arrows, the x offsets are reversed (+10 instead of -10).

#### Return message (dashed line, filled arrowhead):
```
B --> A : response
```
```xml
<polygon fill="#181818" points="39.3618,93.1094,29.3618,97.1094,39.3618,101.1094,35.3618,97.1094"
         style="stroke:#181818;stroke-width:1;"/>
<line style="stroke:#181818;stroke-width:1;stroke-dasharray:2,2;" x1="33.3618" x2="113.6558"
     y1="97.1094" y2="97.1094"/>
```
Dashed pattern for return messages: `stroke-dasharray:2,2`.

#### Thin arrow (open arrowhead -- two separate lines, no polygon):
```
A ->> B : thin
```
```xml
<line style="stroke:#FF0000;stroke-width:1;" x1="192.8174" x2="182.8174" y1="126.4199" y2="122.4199"/>
<line style="stroke:#FF0000;stroke-width:1;" x1="192.8174" x2="182.8174" y1="126.4199" y2="130.4199"/>
<line style="stroke:#FF0000;stroke-width:1;" x1="28.3618" x2="193.8174" y1="126.4199" y2="126.4199"/>
```
Two short lines from the tip (x1,y1) going back 10px at +/-4px vertical offset, plus the shaft line. **No polygon** for thin arrows.

### 6.2 Class Diagram Relationships

All class relationships use `<path>` for the shaft with Bezier curves:
```xml
<path codeLine="3" d="M27.83,55.26 C27.83,72.94 27.83,91.13 27.83,108.79"
      fill="none" id="A-to-B" style="stroke:#181818;stroke-width:1;"/>
```

Note: The `codeLine` attribute appears on many class link paths. The `id` attribute follows the pattern `{source}-to-{target}` or `{source}-backto-{target}` for reverse links, or `{source}-{target}` for plain associations.

#### Arrow type 0: `-->` (Dependency / solid arrow)
- Path: solid (`stroke:#181818;stroke-width:1;`)
- Head: filled polygon pointing at target
```xml
<polygon fill="#181818" points="27.83,114.79,31.83,105.79,27.83,109.79,23.83,105.79,27.83,114.79"
         style="stroke:#181818;stroke-width:1;"/>
```
**Arrowhead polygon pattern** (pointing down):
```
tip_x,   tip_y        (tip)
tip_x+4, tip_y-9      (right wing)
tip_x,   tip_y-5      (inner notch)
tip_x-4, tip_y-9      (left wing)
tip_x,   tip_y        (close)
```

#### Arrow type 1: `..>` (Dashed dependency)
- Path: dashed (`stroke-dasharray:7,7;`)
- Head: same filled polygon as type 0

#### Arrow type 2: `--` (Association / plain line)
- Path: solid, no dasharray
- Head: **no arrowhead at all**

#### Arrow type 3: `<|--` (Extension / inheritance)
- Path: solid
- Head: **hollow triangle** (unfilled polygon)
```xml
<polygon fill="none" points="27.83,55.26,21.83,73.26,33.83,73.26,27.83,55.26"
         style="stroke:#181818;stroke-width:1;"/>
```
Triangle pattern (pointing up, at source):
```
tip_x,   tip_y        (tip)
tip_x-6, tip_y+18     (left base)
tip_x+6, tip_y+18     (right base)
tip_x,   tip_y        (close)
```

#### Arrow type 4: `..|>` (Dashed extension / implements)
- Path: dashed (`stroke-dasharray:7,7;`)
- Head: hollow triangle (same as type 3 but at target end)

#### Arrow type 5: `*--` (Composition)
- Path: solid
- Head: **filled diamond** at source
```xml
<polygon fill="#181818" points="27.83,55.26,23.83,61.26,27.83,67.26,31.83,61.26,27.83,55.26"
         style="stroke:#181818;stroke-width:1;"/>
```
Diamond pattern:
```
tip_x,   tip_y        (top point)
tip_x-4, tip_y+6      (left point)
tip_x,   tip_y+12     (bottom point)
tip_x+4, tip_y+6      (right point)
tip_x,   tip_y        (close)
```

#### Arrow type 6: `o--` (Aggregation)
- Path: solid
- Head: **hollow diamond** at source (same shape as composition but `fill="none"`)
```xml
<polygon fill="none" points="27.83,55.26,23.83,61.26,27.83,67.26,31.83,61.26,27.83,55.26"
         style="stroke:#181818;stroke-width:1;"/>
```

#### Arrow type 7: `<-->` (Bidirectional dependency)
- Path: solid
- Head: **filled arrowheads at BOTH ends**

#### Arrow type 8: `<..>` (Bidirectional dashed dependency)
- Path: dashed (`stroke-dasharray:7,7;`)
- Head: filled arrowheads at both ends

### 6.3 State/Activity/Component Links

Same as class links -- `<path>` with Bezier curves and `<polygon>` arrowheads. Style pattern:
```xml
<path d="M45.65,26.26 C45.65,39.95 45.65,60.07 45.65,79.52"
      fill="none" id="*start*-to-SS" style="stroke:#181818;stroke-width:1;"/>
<polygon fill="#181818" points="45.65,85.52,49.65,76.52,45.65,80.52,41.65,76.52,45.65,85.52"
         style="stroke:#181818;stroke-width:1;"/>
```

### 6.4 Use Case `<<include>>` / `<<extend>>` links

Dashed lines with labels:
```xml
<path ... style="stroke:#181818;stroke-width:1;stroke-dasharray:7,7;"/>
<polygon fill="#181818" points="..." style="stroke:#181818;stroke-width:1;"/>
<text ... font-size="13">&#171;include&#187;</text>
```

## 7. Processing Instructions

### Opening PI (immediately after `<svg>` open tag):
```xml
<?plantuml 1.2026.3beta6?>
```
Format: `<?plantuml {version}?>`

### Closing PI (last child of top-level `<g>`):
```xml
<?plantuml-src Syp9J4vLqBLJSCfFKh1IA4ejB4qjBk5AJ2x9B4iiIGK9Su6aT85A7C5gGOgA2_BpYbEvKbAHrG40?>
```
Format: `<?plantuml-src {encoded-source}?>`

The encoded source is the diagram text run through Deflate compression then a custom Base64 variant.

## 9. Separator Lines in Class Boxes

Class boxes use horizontal `<line>` elements to separate the name, fields, and methods compartments:

```xml
<!-- After class name -->
<line style="stroke:#181818;stroke-width:0.5;" x1="8" x2="93.1113" y1="39" y2="39"/>
<!-- After fields -->
<line style="stroke:#181818;stroke-width:0.5;" x1="8" x2="93.1113" y1="47" y2="47"/>
```

The `x1` is `rect.x + 1` and `x2` is `rect.x + rect.width - 1` (1px inset from the box edges). The `y` values are computed from the font metrics and padding.

For classes with no fields or methods, both separator lines are still rendered (at the same or adjacent y positions), creating the classic three-compartment look even when empty.

### State diagram separators:
State boxes with descriptions use the same pattern:
```xml
<line style="stroke:#181818;stroke-width:0.5;" x1="7" x2="84.292" y1="112.4883" y2="112.4883"/>
```

## 10. Notes

### Standard note shape (folded corner):

Notes are rendered as a path creating a rectangle with a folded top-right corner:

```xml
<path d="M129.01,6 L129.01,45.27 L94.5,49.27 L129.01,53.27 L129.01,92.5527
         A0,0 0 0 0 129.01,92.5527
         L222.1003,92.5527 A0,0 0 0 0 222.1003,92.5527
         L222.1003,16 L212.1003,6 L129.01,6
         A0,0 0 0 0 129.01,6"
      fill="#FEFFDD" style="stroke:#181818;stroke-width:0.5;"/>
```

The fold triangle at top-right:
```xml
<path d="M212.1003,6 L212.1003,16 L222.1003,16 L212.1003,6"
      fill="#FEFFDD" style="stroke:#181818;stroke-width:0.5;"/>
```

The fold is a 10x10 triangle. The main path goes:
1. Start at top-left
2. Down to the left "pointer" area (if the note connects to something on the left)
3. Continue down to bottom-left
4. Across to bottom-right
5. Up to the fold point (right edge, 10px below top)
6. Diagonal to top-right (fold crease)
7. Back to top-left

### Note with pointer (connected to entity on left):

The path includes a triangular pointer extending to the left. In the example, the note for `MyClass` has a pointer:
```
L129.01,45.27 L94.5,49.27 L129.01,53.27
```
This creates a left-pointing triangle (pointer) on the left edge of the note.

### Standalone notes in sequence diagrams:

Standard rectangle with folded corner:
```xml
<path d="M5,80.7988 L5,105.7988 L78,105.7988 L78,90.7988 L68,80.7988 L5,80.7988"
      fill="#FEFFDD" style="stroke:#181818;stroke-width:0.5;"/>
<path d="M68,80.7988 L68,90.7988 L78,90.7988 L68,80.7988"
      fill="#FEFFDD" style="stroke:#181818;stroke-width:0.5;"/>
```

Pattern for sequence notes:
1. Main body: rectangle minus the fold corner area
2. Fold triangle: `corner_x, top_y` -> `corner_x, top_y+10` -> `right_x, top_y+10` -> close

### Hex-shaped notes:
```xml
<polygon fill="#FEFFDD"
         points="53,257.3516,112,257.3516,122,268.3516,112,280.3516,53,280.3516,43,268.3516,53,257.3516"
         style="stroke:#181818;stroke-width:0.5;"/>
```

### Rect-shaped notes (rnote):
```xml
<rect fill="#FEFFDD" height="23" style="stroke:#181818;stroke-width:0.5;"
      width="64" x="105" y="290.6621"/>
```

### Note colors:
- Default note fill: `#FEFFDD` (very light yellow)
- Note stroke: `#181818`, width 0.5
- Note text: `#000000`, font-size 13

## 14. Link/URL Support

When `[[url]]` is specified on a class, the entity content is wrapped in an `<a>` element:

```xml
<a href="https://example.com" target="_top" title="https://example.com"
   xlink:actuate="onRequest" xlink:href="https://example.com"
   xlink:show="new" xlink:title="https://example.com" xlink:type="simple">
  ... rect, text, etc ...
</a>
```

Attributes on the `<a>` element:
| Attribute | Value |
|-----------|-------|
| `href` | The URL |
| `target` | `_top` |
| `title` | The URL (or tooltip if specified) |
| `xlink:actuate` | `onRequest` |
| `xlink:href` | The URL |
| `xlink:show` | `new` |
| `xlink:title` | The URL (or tooltip) |
| `xlink:type` | `simple` |

Individual methods/fields with links get their own nested `<a>` wrappers.

## 15. Composite State / Nested State

Composite states use a header-body pattern:

```xml
<!-- Header fill path (rounded top) -->
<path d="M19.5,86 L158.5703,86 A12.5,12.5 0 0 1 171.0703,98.5 L171.0703,112.4883
         L7,112.4883 L7,98.5 A12.5,12.5 0 0 1 19.5,86"
      fill="#F1F1F1"/>
<!-- Full border rect -->
<rect fill="none" height="301.7283" rx="12.5" ry="12.5"
      style="stroke:#181818;stroke-width:0.5;" width="164.0703" x="7" y="86"/>
<!-- Separator line -->
<line style="stroke:#181818;stroke-width:0.5;" x1="7" x2="171.0703" y1="112.4883" y2="112.4883"/>
<!-- Title text -->
<text ...>Outer Level</text>
```

The header is a filled path with rounded top corners matching the rect's rx/ry=12.5.
A horizontal line separates the header from the body.
Inner states are then rendered as nested elements within the same parent `<g>`.

## 17. Rendering Order

Elements are rendered in a specific order within the top-level `<g>`:

### Sequence diagrams:
1. Activation bars (background layer -- rendered first, appear behind everything)
2. Combo frame outlines (also background)
3. Participant lifelines
4. Participant heads (boxes at top)
5. Participant tails (boxes at bottom)
6. Activation bars (foreground -- rendered again to appear on top)
7. Messages with arrows and labels
8. Notes

Note: Activation bars appear twice in the SVG -- once near the top (behind lifelines) and once after participants (in front). This ensures proper z-ordering.

### Class diagrams:
1. HTML comments identifying each element
2. Entity `<g>` groups (each with rect, icon, text, separator lines)
3. Link `<g>` groups (paths and arrowheads)

### State diagrams:
1. Composite state containers (header + border)
2. Inner state entities
3. Pseudo-states (start, end)
4. Links

## 18. Numeric Precision

PlantUML outputs coordinates with up to 4 decimal places (e.g., `25.5352`, `67.7988`).
Width/height in the root SVG use integer pixel values with `px` suffix.
Internal coordinates are floating-point without units.

## 19. Warning/Error Boxes

Deprecation warnings appear as colored rectangles:
```xml
<rect fill="#FFFFCC" height="16.6406" rx="2.5" ry="2.5"
      style="stroke:#FFDD88;stroke-width:3;" width="533.7842" x="13" y="13"/>
<text fill="#000000" font-family="monospace" font-size="10" ...>This syntax is deprecated...</text>
```

Syntax error SVGs have:
- `background:#000000` in the root style
- Green text (`fill="#33FF02"`)
- Red error text (`fill="#FF0000"`)
- The error display is a version banner + source lines + error message
- No `data-diagram-type` attribute on the root `<svg>`

## 20. Empty `<defs/>` Element

PlantUML always emits an empty `<defs/>` element immediately after the opening processing instruction. No gradients, filters, or reusable definitions are placed here in the default theme.

## 21. Attribute Ordering Convention

PlantUML consistently orders attributes alphabetically within each element. For example, on `<rect>`:
```
fill, height, rx, ry, style, width, x, y
```

On `<text>`:
```
fill, font-family, font-size, [font-style], [font-weight], lengthAdjust, textLength, [text-decoration], x, y
```

On `<svg>`:
```
xmlns, xmlns:xlink, contentStyleType, [data-diagram-type], height, preserveAspectRatio, style, version, viewBox, width, zoomAndPan
```

This alphabetical ordering is consistent across all diagram types. Matching it exactly is important for golden test comparison.
